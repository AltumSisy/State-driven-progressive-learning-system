## 一、心智模型构建

### 1.1 核心概念网络

| 概念 | 定义 | 依赖关系 | 熟练度 |
|------|------|----------|--------|
| **协作悖论** | 使用率60% vs 放手率0-20%，悖论点：流程兜底不足而非能力不够 | → 验收前提 + 举手阈值 + 回滚机制 | ✅ L2 |
| **编排者** | 从implementer到orchestrator：任务编排、拆分（输入输出边界验收）、版本控制合并、人工介入点 | → 协作协议 + 监督规模化 | ✅ L2 |
| **SDLC被压扁** | 流程重叠：瀑布式→并行，边做边测边补文档，周期周→小时 | → 双面性：反馈快 + 错误规模变大 | ✅ L2 |
| **监督规模化** | 审计≠审批；两层监督：常规审计自动化（AI）+ 核心业务人review | → review what matters | ✅ L2 |
| **协作协议** | 5要素：角色、输入、输出、同步点、验收 | → 多智能体协作前提 | ✅ L2 |

### 1.2 专家视角

#### 专家共识

| 议题 | 共识内容 |
|------|----------|
| **协作悖论本质** | AI能力提升不代表放手率提升；放手前提是流程能兜底，不是AI能做 |
| **编排者定位** | 工程师贡献从代码量变成吞吐与质量；不是"写提示词"，而是"把工程化写进流程" |
| **SDLC变化** | 2026年变化不是"IDE里多了助手"，而是流程被压扁：需求/实现/测试/文档重叠 |
| **监督转变** | 从review everything到review what matters；人类注意力只花在关键点 |

#### 专家分歧

| 议题 | 不同观点 |
|------|----------|
| **放手率提升路径** | 有人认为等AI能力更强，有人认为要先补流程设计（验收+举手+回滚） |
| **编排者门槛** | 有人认为需要工程师背景，有人认为产品经理也能做编排 |

### 1.3 深度测试问题

> **Q1**: 为什么60%用AI但只有0-20%放手？
>
> **预期理解层级**：
> - L0（表面）：能力不够，不敢放手
> - L1（关联）：需要流程设计
> - L2（深层）：悖论本质——放手前提不是"AI能做"，而是"流程能兜底"；使用率提升≠放手率提升，违反直觉

> **Q2**: SDLC"压扁"压的是什么？
>
> **预期理解层级**：
> - L0（表面）：周期变短
> - L1（关联）：流程重叠
> - L2（深层）：压的是流程边界——瀑布式→并行，需求/实现/测试/文档重叠；周期周→小时；双面性：反馈快（好）+ 错误规模变大（坏）

> **Q3**: 监督规模化不是"不review"而是什么？
>
> **预期理解层级**：
> - L0（表面）：减少人的工作量
> - L1（关联）：自动化审计
> - L2（深层）：把监督拆成两层——常规审计自动化（格式/静态检查/单测）+ 人类聚焦高风险（业务风险/契约风险）；从review everything到review what matters

### 3.2 反事实情境

> **情境1**：团队AI使用率80%，放手率40%。说明协作悖论解决得好吗？
>
> **L2答案**：悖论被打破了。打破的前提：验收门槛前置（测试/lint/权限）、举手阈值明确（改权限/账务/公共API必须同步）、回滚机制准备（一键熔断）、阶段性验收点。放手率40%说明把"敢放手的前提"都补齐了。

> **情境2**：工程师说"我主要做验收，代码都是AI写的，我算是编排者了。"算不算真正的编排者？
>
> **L2答案**：不算。编排者四要素：任务编排（并行/顺序）、任务拆分（输入输出边界验收）、版本控制合并策略、人工介入点安排。只做验收不够，必须能设计流程、分配任务、设验收门槛。

> **情境3**：团队把所有常规审计自动化了，但没有设计"举手阈值"。有什么问题？
>
> **L2答案**：两类错误会被扩大：业务风险错误（改权限/账务/资金，AI做完才发现）+ 接口契约错误（改公共API，其他服务不知道，集成返工）。举手阈值作用：把风险前置到决策点，不堆到review点。

## 四、补充知识：原文八大趋势与四优先级

### 八大趋势

| 趋势 | 核心判断 | 工程含义 |
|------|----------|----------|
| 01 SDLC被压扁 | 需求/实现/测试/文档重叠，周期周→小时 | 验收门槛必须前置 |
| 02 工程师变编排者 | 从implementer到orchestrator | 工程化写进流程，不是写进聊天框 |
| 03 多智能体协作 | 单智能体不够用，协作协议是门槛 | 先写协作协议，再让智能体开工 |
| 04 长时运行智能体 | 任务跨度几分钟→几天/几周 | 状态管理、回滚、验收成刚需 |
| 05 监督规模化 | review everything→review what matters | 审计自动化+人类聚焦高风险 |
| 06 每个人更全栈 | 相邻领域执行门槛降低 | 智能体打穿边界，但治理要跟上 |
| 07 经济学变化 | 27%工作本来不会被做 | 多产出而非省时间 |
| 08 安全双向加速 | 同样能力增强防御与进攻 | 安全架构前置到智能体系统设计 |

### 四个优先级

| 优先级 | 内容 | 行动 |
|--------|------|------|
| 1 | 掌握多智能体协作 | 用编排解决复杂度，不只靠更大模型 |
| 2 | 把监督规模化 | 自动化审计+升级机制，人只看关键点 |
| 3 | 扩展到工程之外 | 让领域专家自己解决，但必须在平台护栏里 |
| 4 | 安全架构前置 | 从系统设计早期就把权限/审计/隔离/回滚写进去 |

## 五、延伸概念

### 5.1 举手阈值规则

| 操作类型 | 阈值规则 | 原因 |
|----------|----------|------|
| 改权限/账务/资金/合规 | **必须举手** | 业务风险最高 |
| 改公共API/schema | **先举手对齐方案** | 接口契约影响其他服务 |
| 改可验证小逻辑 | **先做再提验收报告** | 低风险，可快速回归 |

### 5.2 监督两层拆分

| 层 | 内容 | 执行者 |
|----|------|--------|
| 常规审计自动化 | 格式检查、静态检查、单测、依赖风险扫描、明显漏洞检测、风格一致性 | AI |
| 人类注意力聚焦 | 高风险diff、边界条件、策略决策、不确定点、业务风险判断 | 人 |

### 5.3 协作协议5要素

| 要素 | 内容 |
|------|------|
| **角色** | 做什么，谁不做什么（尤其是"谁能改公共文件"） |
| **输入** | 资料范围、约束条件、禁止项 |
| **输出** | 要交付什么（补丁、PR、设计文档、风险清单、测试报告） |
| **同步点** | 什么时候停下来对齐接口（先产出API.md或interface.ts） |
| **验收** | 一条可执行的验收命令或检查清单 |
