## 一、心智模型：核心概念网络（对应README 1.1）

```mermaid
mindmap
    root((Agentic Coding))
        协作悖论
            数据：60%使用 vs 0-20%放手
            悖论点：流程兜底不足
            不是能力不够
            原因：验收前提+举手阈值+回滚机制
            结果：编排者角色
        编排者
            四要素
                任务编排（并行/顺序）
                任务拆分（输入输出边界验收）
                版本控制合并策略
                人工介入点安排
            贡献：吞吐与质量
            不是写提示词
        SDLC被压扁
            本质：流程重叠
            瀑布式→并行
            边做边测边补文档
            周期：周→小时
            双面性
                反馈快（好）
                错误规模变大（坏）
        监督规模化
            审计≠审批
                审计=技术检查
                审批=业务开关
            两层监督
                常规审计自动化（AI）
                核心业务人review（人）
            核心转变
                review everything→review what matters
        协作协议
            五要素
                角色：做什么不做什么
                输入：范围约束禁止项
                输出：交付什么
                同步点：何时对齐
                验收：可执行命令
```

## 二、深度测试问题（对应README 1.3）

```mermaid
mindmap
    root((**深度测试**))
        **Q1: 为什么60用0放**
            L0: 能力不够不敢放手
            L1: 需要流程设计
            L2答案:
                悖论本质：流程兜底不足
                放手前提：验收前提+举手阈值+回滚机制
                使用率提升≠放手率提升
        **Q2: SDLC压扁压什么**
            L0: 周期变短
            L1: 流程重叠
            L2答案:
                压的是流程边界
                瀑布式→并行
                需求/实现/测试/文档重叠
                双面性：反馈快+错误扩
        **Q3: 监督规模化不是什么**
            L0: 减少人的工作量
            L1: 自动化审计
            L2答案:
                拆成两层
                常规审计自动化（AI）
                人类聚焦高风险（人）
                review everything→what matters
```

## 三、对抗测试（对应README 3.1-3.2）

```mermaid
mindmap
    root((对抗测试))
        脆弱点诊断
            协作悖论只说结果
                风险: 中
                补救: 悖论数据+悖论点
            SDLC压扁=风险提高
                风险: 高
                补救: 流程重叠+双面性
            监督规模化用审批
                风险: 中
                补救: 审计≠审批
            举手阈值风险笼统
                风险: 中
                补救: 两类具体风险
        反事实情境
            情境1：使用率80%放手率40%
                L2答案: 悖论被打破
                    前提补齐：验收+举手+回滚
            情境2：只做验收自称编排者
                L2答案: 不算编排者
                    缺四要素：编排/拆分/版本控制/介入点
            情境3：审计自动化无举手阈值
                L2答案: 两类错误扩大
                    业务风险+契约风险
                    阈值作用：风险前置到决策点
```

## 四、工程视角：八大趋势（对应README 四）

```mermaid
mindmap
    root((八大趋势))
        01 SDLC被压扁
            流程重叠
            周期周→小时
            行动: 验收门槛前置
        02 编排者
            implementer→orchestrator
            行动: 工程化写进流程
        03 多智能体协作
            协作协议是门槛
            行动: 先写协作协议
        04 长时运行
            几分钟→几天/几周
            行动: 状态管理+回滚
        05 监督规模化
            review what matters
            行动: 审计自动化
        06 全栈扩展
            相邻领域门槛降低
            行动: 治理护栏跟上
        07 经济学变化
            27%工作本来不会被做
            行动: 多产出而非省时间
        08 安全双向
            防御与进攻同时增强
            行动: 安全架构前置
```

## 五、工程视角：四个优先级（对应README 四）

```mermaid
mindmap
    root((四优先级))
        优先级1：多智能体协作
            用编排解决复杂度
            不只靠更大模型
        优先级2：监督规模化
            自动化审计+升级机制
            人只看关键点
        优先级3：扩展到工程之外
            领域专家自己解决
            必须在平台护栏里
        优先级4：安全架构前置
            权限/审计/隔离/回滚
            写进系统设计早期
```

## 六、工程视角：举手阈值规则（对应README 五.1）

```mermaid
mindmap
    root((举手阈值))
        高风险：必须举手
            改权限
            改账务
            改资金
            改合规逻辑
            原因: 业务风险最高
        中风险：先举手对齐
            改公共API
            改schema
            改跨服务契约
            原因: 接口契约影响其他服务
        低风险：先做再报告
            改可验证小逻辑
            快速回归bug
            原因: 低风险可验证
```
