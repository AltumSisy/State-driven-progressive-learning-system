# L-SKILLS: Claude Skills 入门 - 快速复习

> **格言**: "把方法做成资产，用时按需加载"

## 核心概念速记

| 概念 | 一句话定义 | 记忆口诀 |
|------|-----------|----------|
| Skills | 程序化知识封装，可版本化工程资产 | "怎么做固化成资产" |
| 渐进式披露 | 发现→激活→执行，分层加载 | "三段加载不一次性" |
| description | 路由规则：何时用+产出+触发词 | "写太泛变死文件" |
| 脚本化 | 确定性交给程序，双指标提升 | "不进上下文稳+省" |
| 验收标准 | 可执行验证，不只看表面 | "能测试才算验收" |

---

## 核心原则（面试必答）

### 1. Skills ≠ Prompt 的三个关键差异

1. **渐进式披露**：分层加载，不占常驻上下文
2. **脚本执行能力**：把确定性交给程序
3. **可版本化工程资产**：可验收、可团队共享

### 2. 设计三要素

| 要素 | 位置 | 内容 |
|------|------|------|
| 流程 | SKILL.md正文 | 按顺序做什么 |
| 边界 | SKILL.md正文 | 什么时候不做 |
| 验收 | SKILL.md正文 | 什么叫完成（可验证） |

### 3. 脚本不进上下文的双指标

- **稳定性**：确定性部分交给程序
- **成本可控**：脚本逻辑不塞进prompt

## 常见陷阱

| 陷阱 | 错误理解 | 正确理解 |
|------|----------|----------|
| 渐进式披露 | "触发后全部暴露" | 发现→激活→执行三层加载 |
| description写太泛 | "触发率低" | 路由规则失效→变成死文件 |
| 验收=确定性 | "验收保证确定性" | 验收解决输出正确性，确定性是脚本化解决的 |
| 入口文件长 | "内容完整就好" | 入口短（500行）+ 细节拆分引用 + 单一工作流 |

## 非确定性流程验证四种方案

| 方案 | 核心思想 | 适用场景 |
|------|----------|----------|
| 输出结构化 | 验证结构不验证内容 | 报告类Skill |
| 中间检查点 | 每步有验收标准 | 多步骤流程 |
| Hook拦截 | Harness层面防护 | 安全敏感操作 |
| 人工确认 | 不确定性交给用户 | 关键决策点 |

## 自测问题

1. **为什么Skills不是"更高级的提示词"？**
   <details>
   <summary>点击查看答案</summary>
   
   L2答案：因为加入了脚本执行能力，把确定性交给程序，把不确定性留给模型，形成可版本化、可验收的工程资产。三层关键：渐进式披露（不占常驻上下文）+ 脚本能力（双指标提升）+ 工程资产（可版本化）。
   </details>

2. **description写太泛会发生什么？**
   <details>
   <summary>点击查看答案</summary>
   
   L2答案：路由规则失效，Skill变成"永远不会被触发的死文件"。必须写清三要素：何时使用 + 会产出什么 + 触发词。
   </details>

3. **非确定性流程如何验证？**
   <details>
   <summary>点击查看答案</summary>
   
   L2答案：四种方案：输出结构化（验证结构不验证内容）、中间检查点（每步有验收标准）、Hook拦截（Harness层面防护）、人工确认（关键决策点交给用户）。
   </details>

## 生命周期管理速记

### 四步闭环

| 步骤 | 内容 |
|------|------|
| 宣贯本质 | 资产工程化 + 重复性验收 + 30分钟行动 |
| 分级落盘 | P0-P3分级 + GitLab + PR评审 |
| 落地检查 | 8项清单 + Agent Review + PR Review |
| 债务治理 | 5种债务 + 月度盘点 + 持续迭代 |

### 分级标准

| 级别 | 影响 | 验收要求 |
|------|------|----------|
| P0 | 代码/系统变更 | 测试命令必须通过 |
| P1 | 系统稳定性 | 复盘草稿完整 |
| P2 | 团队协作 | 字段完整即可 |
| P3 | 效率提升 | 格式符合标准 |

### 命中率等级

| 命中率 | 状态 | 行动 |
|--------|------|------|
| ≥80% | 健康 | 继续维护 |
| 50-80% | 关注 | 优化description |
| <50% | 死文件 | 修复或删除 |

### 债务类型

| 类型 | 指标 | 时限 |
|------|------|------|
| 验收债务 | 不通过率≥20% | P0/P1 24h |
| 触发债务 | 命中率<50% | 7天 |
| 重复债务 | 重叠数≥2 | 14天 |
| 过期债务 | >90天 | 30天 |
| 错误债务 | 错误率≥10% | P0/P1 24h |

### 自测问题

4. **Skill如何分级？**
   <details>
   <summary>点击查看答案</summary>
   
   L2答案：按最坏情况影响判断。P0影响代码/系统变更（测试必须通过），P1影响系统稳定性（复盘完整），P2影响团队协作（字段完整），P3辅助提效（格式符合）。
   </details>

5. **命中率<50%意味着什么？**
   <details>
   <summary>点击查看答案</summary>
   
   L2答案：Skill变成死文件，路由规则失效。需要优化description（添加触发词、扩展范围）或删除。
   </details>

6. **5种Skill债务是什么？**
   <details>
   <summary>点击查看答案</summary>
   
   L2答案：验收债务（输出不满足标准）、触发债务（命中率低）、重复债务（场景重叠）、过期债务（流程变更未更新）、错误债务（产生错误输出）。P0/P1的验收/错误债务24小时内修复。
   </details>
