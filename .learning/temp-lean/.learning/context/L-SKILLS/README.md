## 一、心智模型构建

### 1.1 核心概念网络

| 概念 | 定义 | 依赖关系 | 熟练度 |
|------|------|----------|--------|
| **Skills** | 程序化知识封装：把"怎么做"固化成可版本化、可按需加载的工程资产 | → 渐进式披露 + 脚本化 | ✅ L2 |
| **渐进式披露** | 分层加载机制：发现→激活→执行，不一次性塞进上下文 | → name/description → SKILL.md → references/scripts | ✅ L2 |
| **description** | 路由规则：写清"何时使用+会产出什么+触发词"，决定Skill是否被调用 | → 触发词匹配 | ✅ L2 |
| **脚本化** | 把确定性部分交给程序执行，模型只负责编排与判断 | → 稳定性 + 成本可控 | ✅ L2 |
| **验收标准** | 可执行验证：测试命令、格式校验、输出字段、示例对比 | → 不只"看起来对" | ✅ L2 |

### 1.2 专家视角

#### 专家共识

| 议题 | 共识内容 |
|------|----------|
| **Skills定位** | 不是"更高级的提示词"，而是"方法论做成资产"。解决的是"怎么做才稳定"，不是"知道什么"。 |
| **加载机制** | 渐进式披露是核心工程价值：能力规模可很大，但常驻上下文成本可控。 |
| **验收标准** | 验收必须可执行：测试命令、格式校验、输出字段、示例对比。不能只"看起来对"。 |
| **脚本价值** | 最被低估的部分：脚本不进上下文，模型只调用它读输出，直接提升稳定性。 |

#### 专家分歧

| 议题 | 不同观点 |
|------|----------|
| **入口文件长度** | 文章建议500行以内，但实际复杂流程可能需要更长。折中方案：入口短，细节拆分引用。 |
| **团队落地路径** | 文章强调"沉淀库"，但实际团队可能先从个人技巧合集开始，逐步规范化。 |

### 1.3 深度测试问题

> **Q1**: 为什么 Skills 不是"更高级的提示词"？
>
> **预期理解层级**：
> - L0（表面）：因为放在文件里而不是对话中
> - L1（关联）：因为分层加载，不占用常驻上下文
> - L2（深层）：因为加入了脚本执行能力，把确定性交给程序，把不确定性留给模型，形成可版本化、可验收的工程资产

> **Q2**: `description` 写得太泛会发生什么？
>
> **预期理解层级**：
> - L0（表面）：模型不知道什么时候用
> - L1（关联）：触发率低，Skill可能不会被调用
> - L2（深层）：description是Skill的路由规则，必须写清"何时使用+会产出什么+触发词"，否则Skill变成"永远不会被触发的死文件"

> **Q3**: 为什么说脚本化是 Skills 最被低估的部分？
>
> **预期理解层级**：
> - L0（表面）：可以执行一些命令
> - L1（关联）：减少模型工作量
> - L2（深层）：脚本不进上下文，模型只调用并读输出，直接提升两个指标：稳定性（确定性部分交给程序）+ 成本可控性（不需要把脚本逻辑塞进prompt）

## 三、对抗测试

### 3.1 脆弱点诊断

| 脆弱点 | 来源问题 | 风险等级 | 补救措施 |
|--------|----------|----------|----------|
| 验收 vs 确定性混淆 | Q1 | 中 | 明确验收解决"输出正确性验证"，确定性是脚本化解决的 |
| 入口文件设计原则 | Q3 | 中 | 记住：入口短（500行）+ 细节拆分引用 + 单一工作流聚焦 |
| 渐进式披露理解偏差 | Section A初始 | 已修正 | 三层加载（发现→激活→执行），不一次性暴露 |

### 3.2 反事实情境

> **情境1**：Skill没有验收标准，流程详细，算不算"好的Skill"？
>
> **L2答案**：不算。验收必须可执行，没有验收标准的问题：模型输出"看起来对"但实际可能缺少字段、格式错误、漏步骤。

> **情境2**：用户说"代码审查"，Skill未触发，最可能原因？
>
> **L2答案**：description路由规则失效——写太泛，没有匹配触发词。

> **情境3**：10个提示词放进一个SKILL.md，有什么问题？
>
> **L2答案**：入口太长→关键约束被淹没；违反单一职责→无法组合复用；加载浪费→用户只需要1个任务却加载了10个的上下文。

## 七、延伸应用：生命周期管理

> 基于理解延伸的实践框架，帮助团队量化管理Skill资产。

### 7.1 四步闭环

| 步骤 | 内容 | 核心动作 |
|------|------|----------|
| **宣贯本质** | 资产工程化 + 重复性验收 + 30分钟行动 | 让团队理解"为什么" |
| **分级落盘** | P0-P3分级 + GitLab + PR评审 | 让Skill有归属 |
| **落地检查** | 8项清单 + Agent Review + PR Review | 让验收可执行 |
| **债务治理** | 5种债务 + 月度盘点 + 持续迭代 | 让资产持续健康 |

### 7.2 Skill分级标准

| 级别 | 定义 | 典型场景 | 验收要求 |
|------|------|----------|----------|
| **P0-工程级** | 直接影响代码/系统变更 | PR review、发布流程 | 测试命令必须通过 |
| **P1-故障级** | 影响系统稳定性 | 故障排障、告警处理 | 复盘草稿完整 |
| **P2-进度级** | 影响团队协作 | 周报、项目状态 | 字段完整即可 |
| **P3-辅助级** | 提升效率 | 文档整理、格式转换 | 格式符合标准 |

### 7.3 命中率量化

| 命中率 | 状态 | 行动 |
|--------|------|------|
| **≥80%** | ✅ 健康 | 继续维护 |
| **50-80%** | ⚠️ 需关注 | 检查description是否需要优化 |
| **<50%** | ❌ 死文件 | 优先修复：重写description或删除 |

**公式**：命中率 = Skill触发次数 / 场景总请求数

### 7.4 债务类型

| 债务类型 | 定义 | 量化指标 | 治理时限 |
|----------|------|----------|----------|
| **验收债务** | 输出不满足验收标准 | 不通过率≥20% | P0/P1 24小时 |
| **触发债务** | 命中率低 | 命中率<50% | 7天 |
| **重复债务** | 多个Skill覆盖同一场景 | 重叠数≥2 | 14天 |
| **过期债务** | 流程变更未更新 | >90天 | 30天 |
| **错误债务** | 产生错误输出 | 错误率≥10% | P0/P1 24小时 |

### 7.5 检查周期

| Skill数量 | 命中率检查 | 债务盘点 |
|-----------|------------|----------|
| ≤10 | 每月 | 每月 |
| 10-50 | 每两周 | 每月 |
| ≥50 | 每周 | 每月 |
