# Skill 落地指南

> 基于Fast Learn总结的核心要点，帮助实际编写Skill时快速落地。

---
- **生成时间**：2026-05-21 22:38
## 一、本质定位：一句话记住

**Skill = 把"怎么做"固化成可版本化工程资产**

不是"更高级的提示词"，而是：
- 可版本化（走PR评审）
- 可验收（有可执行标准）
- 可团队共享（沉淀成库）

---

## 二、落地三要素（每个Skill必须有）

| 要素 | 位置 | 内容要求 | 常见错误 |
|------|------|----------|----------|
| **流程** | SKILL.md正文 | 按顺序做什么，每步有输出 | 写成文档站，没有步骤 |
| **边界** | SKILL.md正文 | 什么时候不做/需要用户补充什么 | 没写边界，模型乱执行 |
| **验收** | SKILL.md正文 | 可验证的完成标准 | 只写"输出看起来对" |

**最小模板**：
```yaml
---
name: xxx
description: 何时用 + 会产出什么 + 触发词
---
# 流程（按序）
1. 步骤1 → 输出X
2. 步骤2 → 输出Y

# 边界
- 以下情况不做：[列出]
- 需要用户补充：[列出]

# 验收（可验证）
- 输出包含：[必填字段]
- 格式符合：[标准]
- 测试命令：[可执行]
```

---

## 三、description：路由规则（决定是否触发）

**写法公式**：
```
何时使用 + 会产出什么 + 触发词
```

**好坏对比**：

| 写法 | 结果 |
|------|------|
| ❌ "帮助用户写代码" | 触发率低，不知道什么时候用 |
| ✅ "用于PR评审，输出风险清单+改动建议+测试建议。触发词：代码审查、review、PR" | 路由清晰，能被触发 |

**记住**：写太泛 → Skill变成死文件

---

## 四、入口文件：三个约束

| 约束 | 标准 | 原因 |
|------|------|------|
| **长度** | ≤500行 | 太长→关键约束被淹没 |
| **内容** | 只放流程+边界+验收 | 细节放references/ |
| **聚焦** | 单一工作流 | 多任务混杂→无法复用 |

**拆分策略**：
```
SKILL.md (入口短)
    ├── 流程：3-5步骤
    ├── 边界：3-5条
    └── 验收：可执行标准
    
references/
    ├── checklist.md (详细清单)
    ├── templates.md (输出模板)
    └── examples.md (示例案例)
```

---

## 五、脚本化：最被低估的价值

**核心原则**：脚本不进上下文

| 交给脚本 | 交给模型 |
|----------|----------|
| 格式校验 | 判断做什么 |
| 批量处理 | 排序编排 |
| 测试执行 | 分析推理 |
| 日志提取 | 决策验证 |

**双指标提升**：
- 稳定性：确定性交给程序
- 成本可控：脚本逻辑不塞进prompt

---

## 六、验收标准：必须可执行

**好验收**：
```yaml
验收：
  - 输出包含：影响评估、根因假设（≥2个）、处置方案
  - 格式符合：Markdown结构
  - 测试命令：npm test 通过
```

**坏验收**：
```yaml
验收：
  - 输出看起来合理  ❌ 不可验证
  - 内容完整  ❌ 模糊标准
```

---

## 七、非确定性流程：四种验证方案

| 方案 | 适用场景 | 实现方式 |
|------|----------|----------|
| **输出结构化** | 报告类Skill | 验证字段存在，不验证内容正确性 |
| **中间检查点** | 多步骤流程 | 每步设检查标准 |
| **Hook拦截** | 安全敏感操作 | Harness层面防护 |
| **人工确认** | 关键决策点 | 明确哪些操作需用户确认 |

## 八、落地检查清单

编写Skill时，逐项检查：

| 检查项 | 标准 | ✅/❌ |
|--------|------|-------|
| description三要素 | 何时用+产出+触发词 | |
| 流程有步骤 | 每步有输出 | |
| 边界明确 | 什么时候不做 | |
| 验收可执行 | 能测试/能验证 | |
| 入口长度 | ≤500行 | |
| 细节拆分 | 放references/ | |
| 单一聚焦 | 一个工作流 | |
| 脚本化考虑 | 确定性部分交给脚本 | |

## 九、30分钟行动清单

**第一步**：选一个高频任务（越土越好）
- PR代码评审
- 线上故障排障
- 周报生成

**第二步**：写短入口（流程+边界+验收）
```yaml
---
name: incident-triage
description: 线上故障排障。用于收集证据、定位根因。触发词：故障、排障、root cause
---
# 流程
1. 先止血 → 判断是否需要降级
2. 再取证 → 列出证据清单
3. 再定位 → 给出根因假设

# 边界
- 影响范围未知时，先问用户

# 验收
- 输出包含：影响评估、证据清单、根因假设
```

**第三步**：把确定性脚本化
- 格式校验 → 写脚本
- 测试命令 → 写脚本

**第四步**：用两个真实样例回归
- 能触发？ ✅
- 能按步骤做？ ✅
- 输出能验收？ ✅

## 十、常见坑速查

| 坑 | 表现 | 修正 |
|-----|------|------|
| 入口太长 | 关键约束淹没 | 拆分到references/ |
| description泛 | 触发率低 | 写清三要素 |
| 验收模糊 | 只说"看起来对" | 写可执行标准 |
| 全靠模型推理 | 不稳定 | 脚本化确定性部分 |
| 多任务混杂 | 无法复用 | 拆成多个Skill |

---

## 十一、与其他组件的分工

| 组件 | 放什么 | 不放什么 |
|------|----------|----------|
| **Prompt** | 本次任务临时指挥 | SOP、长期规则 |
| **Projects** | 长期背景文档 | 方法流程 |
| **Skills** | SOP、模板、清单、脚本入口 | 临时指令 |
| **MCP** | 外部系统连接 | 方法论 |

**一句话**：Skills放"怎么做"，MCP放"怎么连"。

---

## 快速记忆口诀

```
三要素：流程、边界、验收
路由写：何时用、产出、触发词
入口短：五百行、拆细节、单一聚焦
脚本化：不进上下文、稳+省双提升
验收行：可测试、可校验、不只看表面
```

## 团队落地12条清单

| 序号 | 动作 |
|------|------|
| 1 | 把"高频重复任务"拉清单，先选1个做试点 |
| 2 | 明确输入、输出、验收标准，不写长文档 |
| 3 | description写成路由规则：触发场景+产出物 |
| 4 | 入口SKILL.md只写流程、边界、验收 |
| 5 | 细节拆进references/，不常驻上下文 |
| 6 | 确定性操作脚本化：校验、抽取、格式化 |
| 7 | 脚本失败模式设计：缺权限/缺依赖要报清楚 |
| 8 | 每个Skill设owner，允许小步迭代 |
| 9 | 建技能目录页，团队知道有哪些能力可调用 |
| 10 | 每月技能盘点：合并重复、删掉不用、补齐验收 |
| 11 | 安全合规前置：能读/能写/能跑什么写清楚 |
| 12 | Skill超过20个，考虑命名规范与分类体系 |
