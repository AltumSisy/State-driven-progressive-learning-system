# Skill 生命周期管理补充指引

> 基于Fast Learn理解的延伸实践，帮助团队量化管理Skill资产。

---


- **生成时间**：2026-05-19 20:14
## 一、Skill分级标准化

### 问题

"工程级别/故障/进度"分类边界不清晰，可能导致：
- 同一Skill被分到多个类别
- 分类重叠，管理混乱
- 检索困难，不知道去哪个类别找

---

### 分级标准定义

| 级别 | 定义 | 典型场景 | 输出特征 | 验收标准 |
|------|------|----------|----------|----------|
| **P0-工程级** | 直接影响代码/系统变更 | PR review、代码审查、发布流程 | 必须有可执行验收（测试命令） | 验收不通过禁止合并 |
| **P1-故障级** | 影响系统稳定性/可用性 | 故障排障、告警处理、回滚决策 | 必须有时间线+根因+处置方案 | 复盘草稿完整 |
| **P2-进度级** | 影响团队协作/信息同步 | 周报、项目状态、会议纪要 | 结构化输出模板 | 字段完整即可 |
| **P3-辅助级** | 提升效率但不直接影响结果 | 文档整理、日志提取、格式转换 | 输出可验证即可 | 格式符合标准 |

---

### 分类边界规则

| 规则 | 说明 | 示例 |
|------|------|------|
| **单一归属** | 一个Skill只属于一个级别 | PR review → P0，不同时属于P2 |
| **优先级冲突时** | 取影响最大的级别 | "故障复盘报告"→P1（故障）而非P2（报告） |
| **新Skill定级** | 按"最坏情况影响"判断 | 代码生成 → P0（可能引入bug） |

---

### 定级决策树

```
新Skill → 是否影响代码/系统变更？
         ├─ 是 → P0-工程级
         └─ 否 → 是否影响系统稳定性？
                  ├─ 是 → P1-故障级
                  └─ 否 → 是否影响团队协作？
                           ├─ 是 → P2-进度级
                           └─ 否 → P3-辅助级
```

---

### 分级管理策略

| 级别 | 管理策略 | Owner要求 | Review要求 |
|------|----------|-----------|------------|
| P0 | 必须PR评审，必须测试验证 | 技术负责人 | 两人review |
| P1 | 必须PR评审，验收标准明确 | 运维/值班负责人 | 一人review |
| P2 | 可快速迭代，宽松管理 | 团队轮值 | 自检即可 |
| P3 | 个人维护，无需正式流程 | 个人 | 无需review |

## 二、命中率量化

### 问题

"description写太泛→命中率低→Skill变成死文件"，但：
- 不知道哪些Skill命中率低
- 不知道是否需要优化description
- 无法量化判断Skill是否"活着"

---

### 命中率定义

```
命中率 = Skill触发次数 / 该类场景总请求数
```

**统计方式**：
| 统计维度 | 说明 |
|----------|------|
| Skill触发次数 | description匹配成功，Skill被加载的次数 |
| 场景总请求数 | 用户请求中属于该Skill场景的次数（可能未被触发） |

---

### 命中率等级

| 命中率 | 状态 | 行动 |
|--------|------|------|
| **≥80%** | ✅ 健康 | 继续维护 |
| **50-80%** | ⚠️ 需关注 | 检查description是否需要优化 |
| **<50%** | ❌ 死文件 | 优先修复：重写description或删除 |

---

### 统计实现方案

**方案A：轻量日志统计**

```yaml
# Skill执行日志
skill_log:
  - skill_name: code-review
    triggered: true
    timestamp: 2026-05-07T10:30
    user_request: "帮我审查这个PR"
```

定期统计：
```bash
# 每月统计
grep "skill_name: code-review" skill_log | grep "triggered: true" | wc -l
# = 触发次数
```

**方案B：Skill元数据扩展**

```yaml
---
name: code-review
description: ...
metadata:
  trigger_count: 45      # 累计触发次数
  last_triggered: 2026-05-07
  created_at: 2026-04-01
  health_status: healthy  # 基于命中率自动计算
---
```

---

### 命中率优化检查

| 检查项 | 问题表现 | 修正方式 |
|--------|----------|----------|
| description触发词缺失 | 用户说"代码审查"未触发 | 添加触发词：代码审查、review、PR |
| description范围太窄 | 只写"PR review" | 扩展：代码审查、diff检查、风险识别 |
| description与同类Skill重叠 | 多个Skill竞争同一场景 | 明确分工，修改description区分 |

---

### 命中率检查周期

| Skill数量 | 检查周期 |
|-----------|----------|
| ≤10 | 每月检查一次 |
| 10-50 | 每两周检查一次 |
| ≥50 | 每周检查一次，优先关注命中率<50%的 |

## 三、债务定义明确

### 问题

"失败"太笼统，不知道：
- Skill为什么失败
- 失败了多少次
- 是否需要修复或删除

---

### 债务类型定义

| 债务类型 | 定义 | 量化指标 | 风险等级 |
|----------|------|----------|----------|
| **验收债务** | 输出不满足验收标准 | 验收不通过率≥20% | 高 |
| **触发债务** | 命中率低，Skill未被使用 | 命中率<50% | 中 |
| **重复债务** | 多个Skill覆盖同一场景 | 场景重叠数≥2 | 中 |
| **过期债务** | 流程变更但Skill未更新 | 最后更新>90天 | 低 |
| **错误债务** | Skill执行产生错误输出 | 错误率≥10% | 高 |

---

### 债务量化公式

```
验收不通过率 = 验收失败次数 / Skill执行次数
命中率 = 触发次数 / 场景总请求数
场景重叠数 = 覆盖同一场景的Skill数量
错误率 = 输出错误次数 / Skill执行次数
```

---

### 债务等级判定

| 债务类型 | 健康 | 需关注 | 债务 |
|----------|------|--------|------|
| 验收债务 | <10% | 10-20% | ≥20% |
| 触发债务 | ≥80% | 50-80% | <50% |
| 重复债务 | 0 | 1 | ≥2 |
| 过期债务 | <30天 | 30-90天 | >90天 |
| 错误债务 | <5% | 5-10% | ≥10% |

---

### 债务治理策略

| 债务类型 | 治理动作 | Owner责任 | 时间限制 |
|----------|----------|-----------|----------|
| 验收债务 | 重写验收标准或修复流程 | P0/P1必须立即修复 | 24小时 |
| 触发债务 | 优化description或删除 | 命中率<50%优先处理 | 7天 |
| 重复债务 | 合理或明确分工 | 明确description区分 | 14天 |
| 过期债务 | 更新流程或标记废弃 | 标记过期，通知owner | 30天 |
| 错误债务 | 修复脚本或流程 | P0/P1必须立即修复 | 24小时 |

---

### 债务检查清单

每月执行：

| 检查项 | 数据来源 | 判定标准 |
|--------|----------|----------|
| 验收不通过统计 | Skill执行日志 | ≥20%标记债务 |
| 命中率统计 | 触发日志 | <50%标记债务 |
| 场景重叠检查 | description对比 | ≥2标记债务 |
| 最后更新时间 | git log | >90天标记债务 |
| 错误输出统计 | 执行日志 | ≥10%标记债务 |

---

### 债务报告模板

```markdown
## Skill债务月报 (2026-05)

### 债务清单
| Skill | 债务类型 | 指标 | 风险 | Owner | 期限 |
|-------|----------|------|------|-------|------|
| code-review | 验收债务 | 25%不通过 | 高 | 张三 | 已过期 |
| incident-triage | 触发债务 | 30%命中率 | 中 | 李四 | 7天内 |

### 本月治理
- 已修复：xxx (验收债务)
- 已删除：yyy (命中率<10%)
- 待处理：zzz (重复债务)

### 建议优先级
1. P0/P1的验收债务/错误债务 → 立即修复
2. 命中率<30% → 评估是否删除
3. 过期>180天 → 标记废弃
```

## 四、完整生命周期流程图

```
创建Skill
    │
    ▼
定级(P0-P3) ─→ 设置Owner ─→ PR Review
    │
    ▼
上线运行
    │
    ├─→ 统计命中率 → <50%? → 触发债务
    ├─→ 统计验收通过率 → <80%? → 验收债务
    ├─→ 检查场景重叠 → ≥2? → 重复债务
    ├─→ 检查最后更新 → >90天? → 过期债务
    ├─→ 统计错误率 → ≥10%? → 错误债务
    │
    ▼
月度债务盘点
    │
    ├─→ 高风险 → 立即修复
    ├─→ 中风险 → 限期修复
    ├─→ 低风险 → 计划修复
    │
    ▼
修复或删除
    │
    ▼
下个周期
```

## 五、落地执行清单

### 首次落地

| 步骤 | 动作 | 输出 |
|------|------|------|
| 1 | 定义分级标准 | P0-P3分类表 |
| 2 | 现有Skill定级 | 每个Skill标记级别 |
| 3 | 设置Owner | 每个Skill指定owner |
| 4 | 扩展元数据 | 添加trigger_count等字段 |

### 日常运行

| 动作 | 频率 | 执行者 |
|------|------|--------|
| 命中率统计 | 每周 | Skill管理员 |
| 债务检查 | 每月 | Skill管理员 |
| P0/P1验收检查 | 每次执行 | 自动化 |
| 过期检查 | 每季度 | Skill管理员 |

### 债务治理

| 触发条件 | 动作 | 时限 |
|----------|------|------|
| P0/P1验收债务 | 立即修复 | 24小时 |
| 命中率<50% | 优化description | 7天 |
| 重复Skill | 明确分工或合并 | 14天 |
| 过期>180天 | 标记废弃 | 30天 |

## 六、指标仪表盘建议

| 指标 | 显示方式 | 健康阈值 |
|------|----------|----------|
| Skill总数 | 数字 | 按团队规模 |
| P0/P1占比 | 百分比 | 30-50% |
| 平均命中率 | 百分比 | ≥60% |
| 债务数量 | 数字 | <10%总数 |
| 本月新增 | 数字 | 按需 |
| 本月修复 | 数字 | ≥债务数 |

```
分级定：P0影响代码/系统，P1影响稳定，P2影响协作，P3辅助提效

债务查：验收不通过、命中低、重复、过期、错误
周期管：周统计、月盘点、季度检查过期
```
