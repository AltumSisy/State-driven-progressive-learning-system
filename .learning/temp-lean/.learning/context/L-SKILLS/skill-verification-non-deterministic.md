# 非确定性流程的Skill验证方案

## 核心问题

**Q**: 如何判断Skill的流程执行问题？非确定性的流程怎么验证？

---

## 概念澄清

| 流程类型 | 特征 | 验证方式 |
|----------|------|----------|
| **确定性流程** | 每次执行结果相同 | 脚本化 → 测试命令/格式校验 |
| **非确定性流程** | 模型每次推理可能不同 | 无法用脚本直接验证内容"正确性" |

**关键洞察**：确定性流程交给脚本验证，非确定性流程验证的是"结构"和"步骤"，而非"内容正确性"。

---

## 四种解决方案

### 方案1: 输出结构化要求

即使流程非确定，输出格式可以确定。

```yaml
# SKILL.md 中要求
输出必须包含：
- 影响评估（必须）
- 根因假设（至少2个）
- 验证步骤（必须）
- 处置方案（必须）
```

**验证方式**：检查输出是否包含必填字段，不检查字段内容是否"正确"。

**适用场景**：报告类、分析类Skill（故障排障、技术方案）

---

### 方案2: 中间检查点

在关键步骤设置检查，不只检查最终输出。

```yaml
# 工作流程（按序）
1) 先止血：判断是否需要降级/回滚（给出判断依据）
   → 检查点：是否有判断依据
2) 再取证：列出必须收集的证据清单
   → 检查点：清单是否完整（对照references/checklist.md）
3) 再定位：给出根因假设
   → 检查点：假设数量≥2
```

**验证方式**：每一步都有明确的检查标准。

**适用场景**：多步骤复杂流程（排障、审查）

---

### 方案3: Hook机制

在关键事件点做验证，不改Skill本身。

```yaml
# Hook配置
before_tool_use:
  - 如果工具是"删除/覆盖/发布" → 必须二次确认
after_tool_use:
  - 如果输出包含"ERROR" → 提醒用户检查
```

**验证方式**：Harness层面拦截，不依赖Skill内容。

**适用场景**：安全敏感/高风险操作

---

### 方案4: 人工确认关键步骤

非确定性流程的某些决策需要人工介入。

```yaml
# SKILL.md 中明确
涉及以下操作时，必须等待用户确认：
- 删除/覆盖文件
- 发布/部署
- 资金相关操作
```

**验证方式**：把不确定性交给用户判断，不完全依赖模型。

**适用场景**：关键决策点，模型不应自动执行

---

## 设计原则总结

| 原则 | 应用场景 | 核心思想 |
|------|----------|----------|
| **输出结构化** | 流程非确定，输出格式可确定 | 验证结构，不验证内容 |
| **中间检查点** | 多步骤流程 | 每步都有验收标准 |
| **Hook拦截** | 安全敏感操作 | Harness层面防护 |
| **人工确认** | 关键决策点 | 不确定性交给用户 |

---

## 与核心概念的关系

| 核心概念 | 非确定性验证的关系 |
|----------|-------------------|
| **验收标准** | 验收的是"输出结构是否完整"，不是"内容是否正确" |
| **脚本化** | 只能验证确定性部分（格式检查），非确定性部分用其他方案 |
| **Hook** | 是Harness层面的验证机制，不进Skill上下文 |
| **渐进式披露** | references/checklist.md 可用于中间检查点对照 |

---

## 理解层级判断

| 层级 | 特征 |
|------|------|
| **L0** | 不知道非确定性流程如何验证 |
| **L1** | 知道输出结构化，但不知道其他方案 |
| **L2** | 知道四种方案 + 设计原则 + 适用场景 + 与核心概念的关系 |
