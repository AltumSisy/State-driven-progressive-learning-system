# 扩展系统设计分析

你贴出的那段代码来自 `pi` 项目（一个 Claude Code 风格的 coding agent）的扩展系统核心类型定义。这个设计展现了极高的软件工程品味——每一处细节都有明确的设计意图。

## 一、整体架构：分层解耦

```
┌─────────────────────────────────────────────────────────────┐
│                    Extension (加载结果)                       │
│  handlers | tools | commands | flags | shortcuts             │
├─────────────────────────────────────────────────────────────┤
│                 ExtensionRuntime (运行时)                     │
│            State + Actions (分阶段填充)                        │
├─────────────────────────────────────────────────────────────┤
│                 ExtensionAPI (扩展编写接口)                    │
│              注册方法 + 动作方法 + 事件订阅                      │
├─────────────────────────────────────────────────────────────┤
│               ExtensionContext (事件处理上下文)                 │
│              UI | Session | Model | Abort                     │
└─────────────────────────────────────────────────────────────┘
```

### 核心洞察：为什么要这样分层？

**问题**：扩展系统有三个截然不同的生命周期阶段：
1. **加载阶段**：执行扩展的 factory 函数，收集注册信息
2. **运行阶段**：响应事件、执行工具、处理命令
3. **上下文阶段**：每次调用时提供当前状态

**设计答案**：每一层对应一个阶段的需求，避免职责混淆。

## 二、ExtensionRuntime 的精妙设计

你最关注的那段代码：

```typescript
/**
 * Full runtime = state + actions.
 * Created by loader with throwing action stubs, completed by runner.initialize().
 */
export interface ExtensionRuntime extends ExtensionRuntimeState, ExtensionActions {}
```

### 设计意图分析

**为什么用 Intersection Type (`extends A, B`) 而不是单一大接口？**

```typescript
// ❌ 如果这样设计会怎样？
interface ExtensionRuntime {
  flagValues: Map<string, boolean | string>;
  sendMessage: SendMessageHandler;
  // ...所有东西混在一起
}

// ✅ 实际设计
interface ExtensionRuntime extends ExtensionRuntimeState, ExtensionActions {}
```

**核心意图**：语义分离 + 分阶段填充

```typescript
// loader.ts - 加载阶段
export function createExtensionRuntime(): ExtensionRuntime {
  const notInitialized = () => {
    throw new Error("Extension runtime not initialized...");
  };
  
  const runtime: ExtensionRuntime = {
    // State 部分 - 加载阶段就有值
    flagValues: new Map(),
    pendingProviderRegistrations: [],
    assertActive,
    invalidate,
    
    // Actions 部分 - 加载阶段是 throwing stubs
    sendMessage: notInitialized,
    sendUserMessage: notInitialized,
    // ...
  };
  return runtime;
}

// runner.ts - 运行阶段
bindCore(actions: ExtensionActions, ...) {
  // 用真实实现替换 stubs
  this.runtime.sendMessage = actions.sendMessage;
  this.runtime.sendUserMessage = actions.sendUserMessage;
  // ...
}
```

**这是高级的设计 taste**：
- **类型层面**：表达"完整 runtime = state + actions"
- **实现层面**：允许分阶段填充，同时保持类型安全
- **安全层面**：未初始化时调用会抛错，而不是 undefined 行为

### State vs Actions 的分离逻辑

```typescript
interface ExtensionRuntimeState {
  // 扩展加载时就有、且会持久存在的状态
  flagValues: Map<string, boolean | string>;
  pendingProviderRegistrations: Array<{ name: string; config: ProviderConfig }>;
  assertActive: () => void;
  invalidate: (message?: string) => void;
  registerProvider: (name: string, config: ProviderConfig) => void;
}

interface ExtensionActions {
  // 需要 core 绑定后才能执行的动态操作
  sendMessage: SendMessageHandler;
  sendUserMessage: SendUserMessageHandler;
  setModel: SetModelHandler;
  // ...
}
```

**区分标准**：
- **State**：加载阶段可用，或需要在加载阶段收集信息
- **Actions**：需要 session/model 等运行时资源才能执行

## 三、Extension 结构的设计品味

```typescript
export interface Extension {
  path: string;           // 声明路径
  resolvedPath: string;   // 实际路径
  sourceInfo: SourceInfo; // 来源信息（用于错误报告）
  
  // 注册项 - 全部用 Map
  handlers: Map<string, HandlerFn[]>;
  tools: Map<string, RegisteredTool>;
  messageRenderers: Map<string, MessageRenderer>;
  commands: Map<string, RegisteredCommand>;
  flags: Map<string, ExtensionFlag>;
  shortcuts: Map<KeyId, ExtensionShortcut>;
}
```

### 为什么全用 Map？

**设计 taste**：统一查询语义 + 唯一性约束

```typescript
// 查询工具
const tool = extension.tools.get(toolName);

// 检查是否存在
if (extension.handlers.has(eventType)) { ... }

// 遍历
for (const [name, tool] of extension.tools) { ... }
```

**对比如果用 Array**：
```typescript
interface Extension {
  tools: RegisteredTool[];  // 需要手动维护唯一性
}
const tool = ext.tools.find(t => t.name === toolName);  // 更慢，语义不清晰
```

### HandlerFn[] 数组的意图

```typescript
handlers: Map<string, HandlerFn[]>
```

**为什么是数组而不是单个函数？**

```typescript
// 多个扩展可以监听同一个事件
pi.on("session_start", handler1);
pi.on("session_start", handler2);

// runner.ts 遍历所有 handler
for (const handler of handlers) {
  const result = await handler(event, ctx);
  if (result?.cancel) return result;  // 早退出模式
}
```

**设计洞察**：事件系统需要支持多订阅者，但某些事件（如 before_xxx）需要支持取消。

## 四、LoadExtensionsResult 的容错设计

```typescript
export interface LoadExtensionsResult {
  extensions: Extension[];  // 成功加载的
  errors: Array<{ path: string; error: string }>;  // 失败的
  runtime: ExtensionRuntime;  // 共享运行时
}
```

**设计 taste**：不因单个扩展失败而崩溃整个系统

```typescript
// loader.ts 实现
for (const extPath of paths) {
  const { extension, error } = await loadExtension(extPath, ...);
  
  if (error) {
    errors.push({ path: extPath, error });  // 收集错误
    continue;  // 继续加载其他
  }
  
  if (extension) {
    extensions.push(extension);
  }
}
```

**返回结构让调用者决定如何处理错误**：
```typescript
const result = await loadExtensions(paths);
result.errors.forEach(e => console.warn(`Failed: ${e.path}`));
// 系统继续运行，只是少了失败的扩展
```

## 五、共享 Runtime 的设计意图

```typescript
export interface LoadExtensionsResult {
  runtime: ExtensionRuntime;  /** Shared runtime - actions are throwing stubs */
}
```

**为什么 Runtime 是共享的而不是每个扩展一个？**

```typescript
// loader.ts
async function loadExtension(extensionPath, cwd, eventBus, runtime) {
  const extension = createExtension(extensionPath, resolvedPath);
  const api = createExtensionAPI(extension, runtime, cwd, eventBus);
  await factory(api);
  // 所有扩展共用同一个 runtime
}
```

**核心意图**：
1. **Flag 值共享**：`runtime.flagValues` 是所有扩展的 flag 默认值池
2. **Action 唯一性**：`sendMessage` 等动作只有一个实现
3. **生命周期同步**：`invalidate()` 可以同时让所有扩展的 ctx 失效

```typescript
// runner.ts - invalidate 所有扩展
invalidate(message?: string): void {
  if (!this.staleMessage) {
    this.staleMessage = message;
    this.runtime.invalidate(message);  // 一次调用，所有扩展都失效
  }
}
```

## 六、ExtensionAPI vs ExtensionContext 的分层

```typescript
// ExtensionAPI - 给扩展编写者用的注册接口
interface ExtensionAPI {
  on(event: "session_start", handler): void;
  registerTool(tool: ToolDefinition): void;
  registerCommand(name: string, options): void;
  sendMessage(message): void;
}

// ExtensionContext - 给事件处理器用的运行时上下文
interface ExtensionContext {
  ui: ExtensionUIContext;
  sessionManager: ReadonlySessionManager;
  model: Model | undefined;
  abort(): void;
}
```

### 为什么要分成两个接口？

**设计 taste**：权限分离 + 生命周期隔离

| 维度 | ExtensionAPI | ExtensionContext |
|------|--------------|------------------|
| 可用时机 | 扩展加载时（一次性） | 每次事件触发时（多次） |
| 主要能力 | 注册（写入系统） | 查询（读取状态） |
| 持有者 | 扩展模块顶层 | 事件处理函数内部 |

**错误预防设计**：

```typescript
// ❌ 如果混在一起，扩展可能会犯这种错误
export default function(pi) {
  pi.model.current;  // undefined! 加载阶段没有 model
  pi.abort();        // throw! 还没运行
}

// ✅ 实际设计 - 类型层面就阻止了
export default function(pi: ExtensionAPI) {
  // pi 没有 model/abort 属性
  
  pi.on("session_start", (event, ctx: ExtensionContext) => {
    // ctx 才有 model/abort，且只在事件触发时可用
    ctx.model;  // OK，有值
  });
}
```

## 七、ExtensionCommandContext 的扩展

```typescript
interface ExtensionCommandContext extends ExtensionContext {
  waitForIdle(): Promise<void>;
  newSession(options?): Promise<{ cancelled: boolean }>;
  fork(entryId, options?): Promise<{ cancelled: boolean }>;
  reload(): Promise<void>;
}
```

**为什么命令需要额外的 context？**

**设计洞察**：命令是用户主动触发的，有更高的权限需求

```typescript
// 事件处理器（被动响应）
pi.on("session_start", (event, ctx: ExtensionContext) => {
  // 只能读取状态、abort 当前操作
  // 不能发起 session 切换
});

// 命令处理器（主动触发）
pi.registerCommand("fork", {
  handler: async (args, ctx: ExtensionCommandContext) => {
    // 可以 waitForIdle、newSession、fork、reload
    await ctx.fork(entryId);
  }
});
```

**权限隔离的设计 taste**：
- 防止事件处理器意外触发 session 切换
- 确保只有用户明确意图（命令）才能执行危险操作

## 八、设计模式总结

### 1. 分阶段填充模式 (Progressive Filling)

```typescript
interface Runtime extends State, Actions {}

// Phase 1: 创建（只有 State）
const runtime = createExtensionRuntime();

// Phase 2: 绑定（填充 Actions）
bindCore(actions) {
  runtime.sendMessage = actions.sendMessage;
}
```

**好处**：在加载阶段就获得类型完整的对象，但调用会抛错

### 2. 权限分层模式 (Capability Stratification)

```
ExtensionAPI (注册)
  ↓
ExtensionContext (查询 + abort)
  ↓
ExtensionCommandContext (session 控制)
```

**好处**：类型层面防止权限滥用

### 3. 共享状态隔离模式

```typescript
// 共享 runtime，隔离 Extension
for (const ext of extensions) {
  const api = createExtensionAPI(ext, sharedRuntime);
  // ext 收集注册项，runtime 共享状态
}
```

**好处**：统一管理（flag/action）+ 独立收集（handler/tool）

### 4. 事件结果类型分离

```typescript
// 不同事件有不同的结果类型
pi.on("session_before_switch", handler): SessionBeforeSwitchResult
pi.on("session_start", handler): void
pi.on("tool_call", handler): ToolCallEventResult
```

**好处**：类型安全 + 文档化

## 九、这个设计能学到什么？

### 1. **接口分离不是过度设计，是安全设计**

把 API 和 Context 分开，不是"多写几个 interface"，而是：
- 在类型层面表达"加载阶段不能访问运行时状态"
- 让错误在编码阶段就被类型系统发现

### 2. **Map vs Array 的选择是语义选择**

`tools: Map<string, RegisteredTool>` 不仅仅是因为快：
- 表达"tool name → tool"的直接映射语义
- 强制 tool name 的唯一性
- 查询代码 `tools.get(name)` 比 `tools.find(t => t.name === name)` 更清晰

### 3. **Intersection Type 的精妙用法**

`ExtensionRuntime extends ExtensionRuntimeState, ExtensionActions`：
- 不是"组合两个接口"这么简单
- 是表达"Runtime 由两部分组成，且两部分在不同阶段填充"
- 配合 throwing stubs 实现"类型完整但行为受限"

### 4. **错误收集而非崩溃**

`LoadExtensionsResult` 返回 `errors` 数组而不是 throw：
- 单个扩展失败不影响整体
- 让调用者决定如何报告错误
- 生产系统的容错品味

### 5. **共享运行时 + 独立 Extension**

所有扩展共用一个 runtime，但各自有独立的 Extension 对象：
- runtime 管理全局状态（flag/action）
- Extension 收集局部注册项
- invalidate() 可以一次让所有扩展失效

## 十、如何应用到自己的系统？

### 判断标准

如果你的系统有以下特征，可以参考这个设计：

1. **多阶段初始化**：需要先收集注册信息，后绑定实现
   - 用 Intersection Type + 分阶段填充

2. **插件系统**：需要隔离权限、防止错误传播
   - 用 LoadExtensionsResult 收集错误
   - 用 API/Context 分离权限

3. **事件驱动**：多个订阅者，部分事件可取消
   - 用 `Map<string, HandlerFn[]>`
   - 用 Result 类型区分可取消事件

4. **运行时替换**：session 切换、reload 等场景
   - 用 invalidate() + assertActive() 防止 stale context
