## 🧩 第一层：核心抽象 - ToolDefinition 接口（真实结构）

### ❓ 核心问题：为什么需要接口？

**反例：没有接口的混乱**

```typescript
// ❌ 每个工具都不一样，没有统一结构
const bashTool = {
  name: 'bash',
  run: (cmd) => executeCommand(cmd),  // 方法名不一样
  help: 'Execute bash commands',       // 属性名不一样
};

const readTool = {
  toolName: 'read',                   // 又是另一个名字
  execute: (path) => readFile(path),  // 方法名又不一样
  desc: 'Read a file',                // 属性名又不一样
};

// 问题：
// 1. 无法统一管理（注册中心不知道如何调用）
// 2. 无法统一验证（参数格式各不相同）
// 3. 无法统一提示（AI 不知道如何使用）
```

**正例：统一接口的优雅（真实源码）**

```typescript
// ✅ 所有工具都遵循相同接口（基于真实源码 types.ts:435-482）
interface ToolDefinition<TParams extends TSchema, TDetails, TState> {
  // === 身份标识 ===
  name: string;           // 工具名称（LLM 调用时使用）
  label: string;          // UI 显示标签
  description: string;    // 给 LLM 的描述
  
  // === 参数定义 ===
  parameters: TParams;    // ✅ 使用 TypeBox Schema（不是 JSON Schema）
  
  // === AI 提示相关 ===
  promptSnippet?: string;        // 提示词片段
  promptGuidelines?: string[];   // 提示词指南
  
  // === 执行控制 ===
  executionMode?: 'sequential' | 'parallel';  // 执行模式
  prepareArguments?: (args: unknown) => Static<TParams>;  // 参数预处理
  
  // ✅ 核心：execute 是直接的方法（不是 operations 字段）
  execute(
    toolCallId: string,
    params: Static<TParams>,
    signal: AbortSignal | undefined,
    onUpdate: AgentToolUpdateCallback<TDetails> | undefined,
    ctx: ExtensionContext,
  ): Promise<AgentToolResult<TDetails>>;
  
  // === 渲染相关 ===
  renderShell?: 'default' | 'self';
  renderCall?: (...) => Component;
  renderResult?: (...) => Component;
}

// 注册中心可以这样统一处理：
function executeTool(definition: ToolDefinition, params: any) {
  // 统一验证（TypeBox）
  const validated = validate(definition.parameters, params);
  
  // 统一调用
  return definition.execute(toolCallId, validated, signal, onUpdate, ctx);
}
```

### ❓ 核心问题：Operations 在哪里？

**之前的错误理解**：

```typescript
// ❌ 错误：我以为 operations 是 ToolDefinition 的字段

  operations: BashOperations;  // ❌ 错误！不存在这个字段
```

**真实的架构（基于 bash.ts）**：

```typescript
// ✅ 正确：Operations 是工厂函数的参数
interface BashToolOptions {
  operations?: BashOperations;  // ✅ 通过 options 传入
  commandPrefix?: string;
  shellPath?: string;
  spawnHook?: BashSpawnHook;
}

function createBashToolDefinition(
  cwd: string,
  options?: BashToolOptions,
): ToolDefinition<typeof bashSchema, BashToolDetails, BashRenderState> {
  // ✅ 关键：在工厂函数内部使用 Operations
  const ops = options?.operations ?? createLocalBashOperations({ shellPath: options?.shellPath });
  const commandPrefix = options?.commandPrefix;
  
  return {
    name: 'bash',
    label: 'bash',
    description: '...',
    parameters: bashSchema,
    
    // ✅ execute 内部调用 ops.exec()
    async execute(toolCallId, params, signal, onUpdate, ctx) {
      const resolvedCommand = commandPrefix ? `${commandPrefix}\n${params.command}` : params.command;
      
      // ✅ 使用 ops.exec() 执行命令
      const result = await ops.exec(resolvedCommand, cwd, {
        onData: (data) => { /* 流式输出 */ },
        signal,
        timeout: params.timeout,
      });
      
      return {
        content: [{ type: 'text', text: output }],
        details: { ... },
      };
    },
  };
}
```

### 🔌 Operations 抽象模式（真实架构）

**BashOperations 接口（bash.ts:40-58）**：

```typescript
/**
 * ✅ Pluggable operations for the bash tool.
 * Override these to delegate command execution to remote systems (SSH, Docker, etc.)
 */
export interface BashOperations {
  exec(
    command: string,
    cwd: string,
    options: {
      onData: (data: Buffer) => void;
      signal?: AbortSignal;
      timeout?: number;
      env?: NodeJS.ProcessEnv;
    },
  ): Promise<{ exitCode: number | null }>;
}
```

**真实的实现**：

```typescript
// ✅ 本地执行实现（bash.ts:66-126）
export function createLocalBashOperations(options?: { shellPath?: string }): BashOperations {
  return {
    exec: async (command, cwd, { onData, signal, timeout, env }) => {
      const child = spawn(shell, [...args, command], {
        cwd,
        detached: process.platform !== 'win32',
        env: env ?? getShellEnv(),
        stdio: ['ignore', 'pipe', 'pipe'],
      });
      
      // 流式输出
      child.stdout?.on('data', onData);
      child.stderr?.on('data', onData);
      
      // 等待进程结束
      const exitCode = await waitForChildProcess(child);
      
      return { exitCode };
    },
  };
}

// ✅ 自定义远程执行实现
class RemoteBashOperations implements BashOperations {
  private ssh: SSHConnection;
  
  constructor(config: SSHConfig) {
    this.ssh = connectSSH(config);
  }
  
  async exec(command, cwd, { onData, signal, timeout }) {
    const stream = await this.ssh.exec(command, { cwd });
    stream.on('data', onData);
    
    const exitCode = await stream.waitForEnd();
    return { exitCode };
  }
}

// ✅ 自定义 Docker 执行实现
class DockerBashOperations implements BashOperations {
  private docker: Docker;
  private container: string;
  
  constructor(container: string) {
    this.docker = new Docker();
    this.container = container;
  }
  
  async exec(command, cwd, { onData, signal, timeout }) {
    const exec = await this.docker.getContainer(this.container).exec({
      Cmd: ['bash', '-c', command],
      AttachStdout: true,
      AttachStderr: true,
    });
    
    const stream = await exec.start();
    stream.on('data', onData);
    
    return { exitCode: await exec.inspect().ExitCode };
  }
}
```

## 🎓 核心设计原则总结

### 1. 单一职责原则
- 每个工具只做一件事
- `read` 只读取，`write` 只写入，`edit` 只编辑

### 2. 开闭原则
- ✅ 对扩展开放：新增 Operations 无需修改工具定义
- ✅ 对修改封闭：ToolDefinition 接口稳定不变

### 3. 依赖倒置原则
- ✅ 工具定义不依赖具体执行（execute 内部调用 ops.exec）
- ✅ Operations 通过工厂函数参数注入

### 4. 接口隔离原则
- 工具集按功能分离（ReadOnly/Coding）
- 用户按需选择，不强制依赖不需要的工具

### 5. 最小知识原则
- 工具之间相互独立
- 不共享全局状态
