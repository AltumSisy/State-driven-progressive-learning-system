# 工具系统设计模式

> 基于 `pi-coding-agent` 源码的真实设计模式分析。理解为什么这样设计，以及如何应用这些模式。

---

## 🎯 四大设计模式概览

| 模式 | 核心思想 | 解决的问题 |
|------|----------|------------|
| **Definition-first** | 先定义，后实例 | 分离静态配置与动态状态 |
| **Operations 抽象** | 接口与实现分离 | 支持多种执行环境 |
| **工厂函数** | 统一创建入口 | 保证一致性、可维护性 |
| **输出截断** | 智能截断策略 | 控制 token 使用，保留关键信息 |

---

## 1️⃣ Definition-first 模式

### 核心问题

> 为什么先创建 Definition，再包装成 AgentTool？

### 真实源码对比

```typescript
// ===== Definition：静态定义（可缓存）=====

const bashDefinition = createBashToolDefinition(cwd, options);
// 返回 ToolDefinition {
//   name: "bash",
//   description: "...",
//   parameters: Type.Object({...}),
//   execute: async (...) => {...},
// }

// ===== AgentTool：动态实例（给 Agent 用）=====

const bashTool = wrapToolDefinition(bashDefinition, () => ctx);
// 返回 AgentTool {
//   name: "bash",
//   description: "...",
//   parameters: {...},
//   execute: (id, params, signal, onUpdate) => 
//     definition.execute(id, params, signal, onUpdate, ctx),  // 注入 ctx
// }
```

### 两者的关键区别

| 维度 | ToolDefinition | AgentTool |
|------|----------------|-----------|
| **性质** | 静态配置 | 动态实例 |
| **可缓存** | ✅ 可缓存、复用 | ❌ 包含运行时状态 |
| **可序列化** | ✅ 元信息可序列化 | ❌ execute 方法无法序列化 |
| **创建成本** | 低 | 高（需要包装、注入 ctx） |
| **用途** | 注册、管理、提示词构建 | Agent 执行调用 |

### 模式优势

## 2️⃣ Operations 抽象模式

### 核心问题

> 同一个工具，如何在不同环境（本地/远程/Docker）执行？

### 真实源码结构

```typescript
// ===== 接口定义：bash.ts:39-57 =====

export interface BashOperations {
  /**
   * Execute a command and stream output.
   * Override these to delegate command execution to remote systems.
   */
  exec: (
    command: string,
    cwd: string,
    options: {
      onData: (data: Buffer) => void;  // 流式输出回调
      signal?: AbortSignal;            // 取消信号
      timeout?: number;                // 超时
      env?: NodeJS.ProcessEnv;         // 环境变量
    },
  ) => Promise<{ exitCode: number | null }>;
}

// ===== 本地实现：bash.ts:65-128 =====

export function createLocalBashOperations(options?: { shellPath?: string }): BashOperations {
  return {
    exec: (command, cwd, { onData, signal, timeout, env }) => {
      // spawn 子进程
      const child = spawn(shell, [...args, command], {
        cwd,
        detached: process.platform !== 'win32',
        env: env ?? getShellEnv(),
        stdio: ['ignore', 'pipe', 'pipe'],
      });
      
      // 流式输出
      child.stdout?.on('data', onData);
      child.stderr?.on('data', onData);
      
      // 等待进程结束
      return waitForChildProcess(child);
    },
  };
}
```

### Operations 注入位置

```typescript
// ===== 工厂函数通过 options 注入：bash.ts:271-278 =====

export function createBashToolDefinition(
  cwd: string,
  options?: BashToolOptions,
): ToolDefinition<typeof bashSchema, BashToolDetails, BashRenderState> {
  // ★ 关键：从 options 获取，没有则用默认本地实现
  const ops = options?.operations ?? createLocalBashOperations({ shellPath: options?.shellPath });
  
  return {
    name: "bash",
    // ...
    async execute(toolCallId, { command, timeout }, signal, onUpdate, ctx) {
      // ★ execute 内部调用 ops.exec()
      await ops.exec(command, cwd, { onData, signal, timeout });
    },
  };
}
```

### 不同环境的实现示例

```typescript
// ===== SSH 远程执行 =====

class RemoteBashOperations implements BashOperations {
  private ssh: SSHConnection;
  
  constructor(config: SSHConfig) {
    this.ssh = connectSSH(config);
  }
  
  async exec(command, cwd, { onData, signal, timeout }) {
    const stream = await this.ssh.exec(command, { cwd });
    stream.on('data', onData);
    
    const exitCode = await stream.waitForEnd();
    return { exitCode };
  }
}

// ===== Docker 容器执行 =====

class DockerBashOperations implements BashOperations {
  private docker: Docker;
  private container: string;
  
  constructor(container: string) {
    this.docker = new Docker();
    this.container = container;
  }
  
  async exec(command, cwd, { onData, signal, timeout }) {
    const exec = await this.docker.getContainer(this.container).exec({
      Cmd: ['bash', '-c', command],
      AttachStdout: true,
      AttachStderr: true,
    });
    
    const stream = await exec.start();
    stream.on('data', onData);
    
    return { exitCode: await exec.inspect().ExitCode };
  }
}
```

## 4️⃣ 输出截断模式

### 核心问题

> LLM 有 token 限制，大文件/大输出如何处理？

### 为什么需要截断

```
问题场景：
├── 读取大文件（100MB log文件）
├── 执行命令产生大量输出（npm install 的日志）
├── grep 匹配太多行（搜索整个项目）
│
└── 如果全部发送给 LLM：
    ├── 超出 context window
    ├── 浪费 token
    └── 丢失关键信息
```

### 真实源码结构

```typescript
// ===== truncate.ts:11-13 常量定义 =====
```

### 三种截断策略

#### ① truncateHead - 保留头部

```typescript
// ===== truncate.ts:78-160 =====

/**
 * Truncate content from the head (keep first N lines/bytes).
 * Suitable for file reads where you want to see the beginning.
 */
export function truncateHead(content: string, options?: TruncationOptions): TruncationResult {
  const maxLines = options?.maxLines ?? DEFAULT_MAX_LINES;
  const maxBytes = options?.maxBytes ?? DEFAULT_MAX_BYTES;
  
  const lines = splitLinesForCounting(content);
  
  // 收集能放下的完整行
  const outputLinesArr: string[] = [];
  let outputBytesCount = 0;
  
  for (let i = 0; i < lines.length && i < maxLines; i++) {
    const line = lines[i];
    const lineBytes = Buffer.byteLength(line, "utf-8") + (i > 0 ? 1 : 0);
    
    if (outputBytesCount + lineBytes > maxBytes) {
      break;  // 字节限制
    }
    
    outputLinesArr.push(line);
    outputBytesCount += lineBytes;
  }
  
  return {
    content: outputLinesArr.join("\n"),
    truncated: true,
    truncatedBy: outputLinesArr.length >= maxLines ? "lines" : "bytes",
    // ...
  };
}
```

```typescript
// ===== truncate.ts:268-276 =====

/**
 * Truncate a single line to max characters, adding [truncated] suffix.
 * Used for grep match lines.
 */
export function truncateLine(line: string, maxChars: number = GREP_MAX_LINE_LENGTH): { text: string; wasTruncated: boolean } {
  if (line.length <= maxChars) {
    return { text: line, wasTruncated: false };
  }
  return { text: `${line.slice(0, maxChars)}... [truncated]`, wasTruncated: true };
}
```

**适用场景**：`grep` 工具匹配的单行过长

```
│ Very long match line with 1000 characters of...     │
└─────────────────────────────────────────────────────┘
                           ↓
┌────────────────────────────────────┐
│ Very long match line with 500 c... │ [truncated]
└────────────────────────────────────┘
```

### 各工具的截断应用

| 工具 | 截断策略 | 原因 |
|------|----------|------|
| `read` | `truncateHead` | 文件开头通常包含重要信息（导入、配置） |
| `bash` | `truncateTail` | 命令输出末尾包含错误、结果 |
| `grep` | `truncateLine` + `head_limit` | 单行过长+匹配数量限制 |
| `find` | `head_limit` | 控制匹配数量 |

### bash 工具的截断实现

```typescript
// ===== bash.ts execute 内部（简化）=====

async execute(...) {
  const output = new OutputAccumulator({ tempFilePrefix: "pi-bash" });
  
  // 流式收集输出
  await ops.exec(command, cwd, {
    onData: (data) => {
      output.append(data);
      // 流式更新 UI
    },
  });
  
  // 截断处理
  const truncation = truncateTail(output.getContent(), {
    maxLines: DEFAULT_MAX_LINES,
    maxBytes: DEFAULT_MAX_BYTES,
  });
  
  // 如果截断，保存完整输出到临时文件
  if (truncation.truncated) {
    const fullPath = output.saveToTempFile();
    return {
      content: [{ type: "text", text: truncation.content }],
      details: {
        truncation,
        fullOutputPath: fullPath,  // ← 用户可以查看完整输出
      },
    };
  }
}
```

## 📋 设计原则总结

### 五大设计原则应用

| 原则 | 在工具系统中的体现 |
|------|-------------------|
| **单一职责** | 每个工具只做一件事：read只读、bash只执行、grep只搜索 |
| **开闭原则** | Operations 抽象：新增执行环境无需修改 Definition |
| **依赖倒置** | 工具定义依赖抽象接口（Operations），不依赖具体实现 |
| **接口隔离** | 工具集按功能分离（ReadOnly/Coding），按需组合 |
| **最小知识** | 工具之间相互独立，不共享全局状态 |
