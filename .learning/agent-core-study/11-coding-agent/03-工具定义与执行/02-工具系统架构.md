# 工具系统架构学习指南

> 本文档专注于工具系统的**架构层面**，不涉及具体工具实现细节。学习如何设计一个可扩展、可维护的工具系统。

---

## 🎯 学习目标

理解 Coding Agent 的工具系统架构，掌握：
- 工具的抽象定义（ToolDefinition 接口）
- 工具的注册与管理（工具注册中心）
- 工具的创建模式（工厂函数）
- 工具的执行抽象（Operations 模式）
- 工具的分类与组合（工具集设计）

---

## 📚 学习路径（按顺序学习）

### 1️⃣ 第一层：核心抽象 - ToolDefinition 接口

**位置**: `tools/index.ts` 或各工具文件的类型定义部分

**核心接口**:
```typescript
interface ToolDefinition<TInput, TDetails> {
  // === 身份标识 ===
  name: string;                  // 工具名称（如 "bash", "read"）
  description: string;           // 工具描述（给 AI 看的）
  
  // === 参数定义 ===
  parameters: JSONSchema;        // 参数 schema（JSON Schema 格式）
  
  // === AI 提示相关 ===
  promptSnippet?: string;        // 提示词片段（如何使用此工具）
  promptGuidelines?: string[];   // 提示词指南（使用注意事项）
  
  // === 执行相关 ===
  operations: TOperations;       // 操作接口（定义如何执行）
  buildDetails?: (input) => TDetails;  // 构建执行上下文
}
```

**关键理解**:
1. **为什么需要接口？**
   - 统一抽象：所有工具遵循相同结构
   - 类型安全：TypeScript 编译期检查
   - 可扩展性：新工具只需实现接口

2. **泛型参数的作用**:
   - `TInput`: 输入参数类型（如 BashInput、ReadInput）
   - `TDetails`: 执行详情类型（如 BashDetails、ReadDetails）

3. **分层设计**:
   - 定义层：`ToolDefinition`（静态，可序列化，可缓存）
   - 实例层：`Tool`（动态，包含运行时状态）
   - 执行层：`Operations`（具体执行逻辑）

**思考问题**:
- ❓ 为什么 `operations` 和 `buildDetails` 是分开的？
- ❓ `promptSnippet` 和 `promptGuidelines` 有什么区别？分别给谁看？

### 2️⃣ 第二层：注册中心 - tools/index.ts

**核心职责**: 统一管理所有工具的创建和注册

**关键函数**:
```typescript
// 单工具创建
createToolDefinition(toolName, cwd, options)  // 根据名字创建定义
createTool(toolName, cwd, options)             // 根据名字创建实例

// 工具集创建
createCodingToolDefinitions(cwd, options)      // 创建写入类工具集
createReadOnlyToolDefinitions(cwd, options)    // 创建只读类工具集
createAllToolDefinitions(cwd, options)         // 创建全工具集
```

**注册表模式**:
```typescript
// 通常实现为映射表
const toolRegistry = {
  'bash': createBashToolDefinition,
  'read': createReadToolDefinition,
  'edit': createEditToolDefinition,
  // ...
};

// 统一创建入口
function createToolDefinition(toolName, cwd, options) {
  const factory = toolRegistry[toolName];
  return factory(cwd, options);
}
```

**关键理解**:
1. **注册表模式的好处**:
   - 解耦：工具定义散落在各文件，注册中心统一管理
   - 动态性：可以根据名字动态创建工具
   - 可扩展：新增工具只需注册到 registry

2. **工具集的分类意义**:
   - 权限控制：只读环境只能用 `ReadOnlyTools`
   - 风险管理：写入类工具需要更严格的权限
   - 模式匹配：某些场景自动选择合适的工具集

**思考问题**:
- ❓ 如果要新增一个工具，需要修改哪些地方？
- ❓ `createToolDefinition` 和 `createTool` 的区别是什么？为什么需要两个函数？

### 3️⃣ 第三层：设计模式

#### 3.1 Definition-first 模式

**模式代码**:
```typescript
// 第一步：创建定义
const definition = createBashToolDefinition(cwd, options);

// 第二步：创建实例
const tool = createBashTool(cwd, options);
```

**为什么先定义再实例化？**

| 维度 | Definition | Instance |
|------|-----------|----------|
| 性质 | 静态配置 | 动态实例 |
| 可缓存 | ✅ 可缓存 | ❌ 不可缓存 |
| 可序列化 | ✅ 可序列化 | ❌ 包含运行时状态 |
| 可共享 | ✅ 多处复用 | ❌ 独立实例 |
| 创建成本 | 低 | 高（可能涉及资源初始化） |

**应用场景**:
- 定义可以被缓存，避免重复创建
- 定义可以被序列化传输（跨进程、跨网络）
- 定义可以被复用，实例是独立的

---

#### 3.2 Operations 抽象模式

**模式代码**:
```typescript
// 定义层：只描述"做什么"
const definition = {
  name: 'bash',
  operations: BashOperations,  // 接口，不是实现
};

// 执行层：描述"怎么做"
interface BashOperations {
  spawn(command: string): Process;
  write(input: string): void;
  kill(): void;
}

// 本地执行实现
createLocalBashOperations(): BashOperations {
  return {
    spawn: (cmd) => child_process.spawn(cmd),
    write: (input) => process.stdin.write(input),
    kill: () => process.kill(),
  };
}

// 远程执行实现（示例）
createRemoteBashOperations(sshConfig): BashOperations {
  return {
    spawn: (cmd) => ssh.exec(cmd),
    write: (input) => ssh.stdin.write(input),
    kill: () => ssh.disconnect(),
  };
}
```

**抽象的好处**:

```
┌─────────────────────────────────────┐
│   Tool Definition (不变)             │
│   - name: 'bash'                     │
│   - operations: BashOperations       │  ← 接口，不依赖具体实现
└─────────────────────────────────────┘
            ↓
┌─────────────────────────────────────┐
│   Operations Implementation (可变)    │
│   ├── LocalBashOperations            │  ← 本地执行
│   ├── RemoteBashOperations           │  ← 远程执行
│   ├── DockerBashOperations           │  ← 容器执行
│   └── SandboxBashOperations          │  ← 沙箱执行
└─────────────────────────────────────┘
```

**关键理解**:
- 同一个工具定义，可以在不同环境执行
- 新增执行环境，无需修改工具定义
- 遵循**开闭原则**：对扩展开放，对修改封闭

**思考问题**:
- ❓ 如果要支持在 Docker 容器中执行命令，需要做什么？
- ❓ Operations 接口应该包含哪些方法？如何平衡灵活性和复杂度？

#### 3.3 工厂函数模式

**模式代码**:
```typescript
// 每个工具都有两个工厂函数
createBashToolDefinition(cwd, options)  // 创建定义
createBashTool(cwd, options)             // 创建实例

createReadToolDefinition(cwd, options)   // 创建定义
createReadTool(cwd, options)             // 创建实例

// ... 所有工具都遵循相同模式
```

**统一的工厂模式**:
```typescript
// 抽象工厂接口
interface ToolFactory {
  createDefinition(cwd: string, options: Options): ToolDefinition;
  createTool(cwd: string, options: Options): Tool;
}

// 具体工厂实现
const BashToolFactory = {
  createDefinition(cwd, options) {
    return {
      name: 'bash',
      description: '...',
      parameters: { ... },
      operations: new LocalBashOperations(options),
    };
  },
  
  createTool(cwd, options) {
    const definition = this.createDefinition(cwd, options);
    return new Tool(definition, options);
  },
};
```

**关键理解**:
1. **一致性**: 所有工具用相同模式创建，降低学习成本
2. **可维护性**: 修改创建逻辑，只需修改工厂函数
3. **可测试性**: 可以 mock 工厂函数进行测试

#### 3.4 输出截断模式

**为什么需要截断？**
- LLM 有 token 限制
- 大文件/大输出会超出上下文
- 需要智能截断，保留关键信息

**截断策略**:
```typescript
// 截断头部（保留尾部）
truncateHead(content, maxSize) {
  return '...[truncated]...' + content.slice(-maxSize);
}

// 截断尾部（保留头部）
truncateTail(content, maxSize) {
  return content.slice(0, maxSize) + '...[truncated]';
}

// 截断中间（保留头尾）
truncateLine(content, maxLines) {
  if (content.lines <= maxLines) return content;
  
  const head = content.slice(0, maxLines / 2);
  const tail = content.slice(-maxLines / 2);
  return `${head}\n...[truncated ${content.lines - maxLines} lines]...\n${tail}`;
}
```

**工具中的应用**:
- `read.ts`: 截断大文件（保留头尾，显示行号）
- `bash.ts`: 截断长输出（保留头尾，显示截断数量）
- `grep.ts`: 限制匹配数量（`head_limit` 参数）

**思考问题**:
- ❓ 为什么不同的工具有不同的截断策略？
- ❓ 如何在截断的同时保留关键信息？

### 4️⃣ 第四层：工具分类与组合

#### 分类逻辑

```
工具分类维度
├── 按权限分类
│   ├── 写入类（Coding Tools）- 可修改文件/执行命令
│   │   ├── bash  - 执行命令
│   │   ├── edit  - 编辑文件
│   │   └── write - 写入文件
│   │
│   └── 只读类（Read-Only Tools）- 仅读取信息
│       ├── read - 读取文件
│       ├── grep - 搜索内容
│       ├── find - 查找文件
│       └── ls   - 列出目录
│
└── 按用途分类（概念性分类）
    ├── 文件操作类
    │   ├── read  - 读取
    │   ├── write - 写入
    │   └── edit  - 编辑
    │
    ├── 搜索查找类
    │   ├── grep - 内容搜索
    │   ├── find - 文件查找
    │   └── ls   - 目录浏览
    │
    └── 命令执行类
        └── bash - 命令执行
```

#### 工具集设计

**三种工具集**:
```typescript
// 1. Coding Tools - 写入类工具集
createCodingToolDefinitions(cwd, options) {
  return [
    createBashToolDefinition(cwd, options),
    createEditToolDefinition(cwd, options),
    createWriteToolDefinition(cwd, options),
    createReadToolDefinition(cwd, options),  // 写入类也需要读取
  ];
}

// 2. Read-Only Tools - 只读类工具集
createReadOnlyToolDefinitions(cwd, options) {
  return [
    createReadToolDefinition(cwd, options),
    createGrepToolDefinition(cwd, options),
    createFindToolDefinition(cwd, options),
    createLsToolDefinition(cwd, options),
  ];
}

// 3. All Tools - 全工具集
createAllToolDefinitions(cwd, options) {
  return [
    ...createCodingToolDefinitions(cwd, options),
    ...createReadOnlyToolDefinitions(cwd, options),
  ];
}
```

**使用场景**:

| 场景 | 工具集 | 原因 |
|------|--------|------|
| 生产环境只读分析 | Read-Only Tools | 防止误修改，降低风险 |
| 开发环境代码编辑 | Coding Tools | 需要写入能力 |
| 完整开发环境 | All Tools | 需要全部能力 |
| CI/CD 流水线 | Read-Only Tools | 安全审计，不修改代码 |

**思考问题**:
- ❓ 为什么 `Coding Tools` 也包含 `read` 工具？
- ❓ 如果要新增一个工具，如何决定它属于哪个工具集？

### 5️⃣ 第五层：参数验证与安全

#### JSON Schema 验证

**每个工具都有参数 schema**:
```typescript
const bashParameters = {
  type: 'object',
  properties: {
    command: {
      type: 'string',
      description: 'The command to execute',
    },
    timeout: {
      type: 'number',
      description: 'Timeout in milliseconds',
      default: 120000,
    },
  },
  required: ['command'],
};
```

**验证流程**:
```
用户输入
  ↓
JSON Schema 验证
  ├─ ✅ 通过 → 执行工具
  └─ ❌ 失败 → 返回错误信息
```

**好处**:
1. **类型安全**: 参数类型在调用前验证
2. **文档化**: Schema 本身就是文档
3. **自动提示**: AI 可以根据 Schema 生成参数

#### 路径安全验证

**常见安全机制**:
```typescript
// 1. 必须使用绝对路径
if (!path.isAbsolute(filePath)) {
  throw new Error('Path must be absolute');
}

// 2. 路径必须在 cwd 内
const resolvedPath = path.resolve(cwd, filePath);
if (!resolvedPath.startsWith(cwd)) {
  throw new Error('Path must be within working directory');
}

// 3. 文件必须存在（对于 read/edit）
if (!fs.existsSync(resolvedPath)) {
  throw new Error('File does not exist');
}

// 4. 必须先 Read 才能 Write/Edit
if (!hasReadFile(resolvedPath)) {
  throw new Error('Must read file before writing/editing');
}
```

**思考问题**:
- ❓ 为什么必须先 Read 才能 Write/Edit？
- ❓ 路径验证为什么要检查 `startsWith(cwd)`？

## 🔑 核心设计原则总结

### 1. 单一职责原则
- 每个工具只做一件事
- `read` 只读取，`write` 只写入，`edit` 只编辑

### 2. 开闭原则
- 对扩展开放：新增工具无需修改现有代码
- 对修改封闭：工具接口稳定不变

### 3. 依赖倒置原则
- 工具定义依赖抽象接口（Operations）
- 不依赖具体实现（Local/Remote/Sandbox）

### 4. 接口隔离原则
- 工具集按功能分离（ReadOnly/Coding）
- 用户按需选择，不强制依赖不需要的工具

### 5. 最小知识原则
- 工具之间相互独立
- 不共享全局状态
